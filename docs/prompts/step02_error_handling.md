# Step 2: エラーハンドリングとログ出力のリファクタリング

## 目的
スプレッドシートの読み込み時（シート名の指定ミス、カラム名の不一致など）に発生するエラーを適切に捕捉し、フロントエンドの画面上およびコンソールログにわかりやすく出力するようにコードをリファクタリングする。

## タスク
1. `src/コード.gs` の改修:
   - スプレッドシートおよび指定したシート（`SHEET_NAME`）が存在するかチェックし、存在しない場合は明確なエラーメッセージを返すこと。
   - 1行目のヘッダーを取得し、要件定義書（`01_requirements.md`）で定義されている必須カラムがすべて存在するか検証（バリデーション）すること。不足しているカラムがあれば、そのカラム名をエラーメッセージに含めること。
   - レスポンスの形式を統一し、成功時は `{ status: "success", data: [...] }`、エラー時は `{ status: "error", message: "エラーの詳細" }` のようなJSON構造でフロントエンドに返すこと。

2. `src/index.html` の改修:
   - GASからのデータ取得処理に対して、エラーハンドリング（`try-catch` やエラーコールバックの処理）を実装すること。
   - データの取得中であることがユーザーに伝わるよう、ローディングUI（「読み込み中...」等のテキストやスピナー）を実装すること。
   - エラー発生時は、画面上にTailwind CSSを用いた視認性の高いエラーメッセージ（赤背景のアラートボックスなど）を表示させること。
   - デバッグが容易になるよう、エラーの詳細情報（例外オブジェクトやバックエンドからのメッセージ）を `console.error` に出力すること。

## 注意事項
- 既存の「全体俯瞰マトリクス」および「月別チェックリスト」のUIコンポーネント、および印刷用CSS（`@media print`）の機能は絶対に破壊しないこと。
- リポジトリのルール（`CLAUDE.md`）に従い、シークレット情報（スプレッドシートID等）は `env.gs` から読み取る構造を必ず維持すること。
